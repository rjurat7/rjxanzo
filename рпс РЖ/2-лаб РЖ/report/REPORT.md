# Отчёт по лабораторной работе №2
**Тема**: «Изучение реализации и расчёта сложности алгоритмов сортировки»  
**Алгоритм (вариант)**: Шейкерная сортировка (Shaker sort)

## Цель работы
Разработать консольное приложение для сортировки целых чисел согласно варианту и оценить сложность полученного алгоритма.

## Задание
- Реализовать выбранный алгоритм сортировки как отдельный модуль (без штатных функций сортировки).
- Реализовать консольное приложение со способами ввода массива:
  - ввод с клавиатуры;
  - генерация случайных целых чисел в диапазоне **от 1 до 99 включительно**;
  - загрузка из файла.
- Отсортировать массив, вывести исходный и отсортированный массивы на экран, предложить выгрузить оба массива в файл.
- Реализовать unit-тесты для модуля сортировки. Тесты должны запускаться отдельно от основного приложения.
- Рассчитать лучшую и худшую временную сложность алгоритма (с подробным объяснением).

## Алгоритм сортировки в виде блок-схемы
Ниже приведена блок‑схема шейкерной сортировки (двунаправленной пузырьковой сортировки).

```mermaid
flowchart TD
    A([Старт]) --> B[Инициализация: left = 0, right = n-1, swapped = true]
    B --> C{left < right<br/>и swapped = true?}
    C -- нет --> Z([Стоп])
    C -- да --> D[swapped = false]
    D --> E[Прямой проход i = left..right-1]
    E --> F{arr[i] > arr[i+1]?}
    F -- да --> G[swap(arr[i], arr[i+1])<br/>swapped = true]
    F -- нет --> H[Следующий i]
    G --> H
    H --> E
    E --> I[right--]
    I --> J[Обратный проход i = right..left+1]
    J --> K{arr[i] < arr[i-1]?}
    K -- да --> L[swap(arr[i], arr[i-1])<br/>swapped = true]
    K -- нет --> M[Следующий i]
    L --> M
    M --> J
    J --> N[left++]
    N --> C
```

## Расчёт лучшей и худшей временной сложности
Обозначим \(n\) — количество элементов массива.

### Что делает один “цикл” шейкера
Алгоритм работает “волнами”:
- **Прямой проход**: сравнивает пары соседних элементов слева направо и проталкивает максимум вправо.
  - Число сравнений порядка \(right-left\).
- Сдвиг границы: `right--`.
- **Обратный проход**: сравнивает пары справа налево и проталкивает минимум влево.
  - Число сравнений порядка \(right-left\).
- Сдвиг границы: `left++`.

Границы сужаются, поэтому длина рабочего диапазона уменьшается примерно на 2 за полный цикл (туда‑обратно).

### Лучшая сложность \(O(n)\)
Если массив уже отсортирован по возрастанию:
- на прямом проходе **не происходит ни одного обмена** → `swapped` остаётся `false`;
- условие `while (left < right && swapped)` сразу прекращает работу после первого прямого/обратного цикла (по факту достаточно одного прямого прохода, но у нас цикл включает и обратную часть; обменов всё равно нет).

Количество сравнений в первом прямом проходе \(\approx n-1\), то есть:
\[
T_{best}(n) = O(n)
\]

### Худшая сложность \(O(n^2)\)
Худший случай для шейкерной сортировки — массив, отсортированный по убыванию:
- почти в каждом сравнении требуется обмен;
- алгоритм делает много проходов, постепенно сужая диапазон.

Оценим число сравнений:
- на первом цикле: \((n-1) + (n-2)\)
- на втором: \((n-3) + (n-4)\)
- и т.д.

Суммарно это порядка суммы:
\[
(n-1) + (n-2) + (n-3) + \dots + 1 = \frac{n(n-1)}{2}
\]
То есть:
\[
T_{worst}(n) = O(n^2)
\]

### Память
Сортировка выполняется “на месте” (только несколько переменных‑индексов), поэтому:
\[
S(n) = O(1)
\]
