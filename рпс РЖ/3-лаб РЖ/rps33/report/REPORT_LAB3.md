# Отчёт — лабораторная работа №3 (шаблон для `rps33`)

**Тема:** «Изучение реализации графических пользовательских интерфейсов и интеграции базы данных…»

## 1. Цель работы

Разработка графического пользовательского интерфейса для приложения, работа с базой данных (хранение исходных и отсортированных массивов), интеграционное тестирование слоя БД отдельным проектом.

## 2. Задание (сводка требований)

- **Git:** ветка `develop-lab3` (выполняется в репозитории; в отчёте укажите факт создания).
- **БД:** СУБД на выбор; структура — хранение целочисленных массивов (до и после сортировки).
- **Интеграционные тесты (отдельно от GUI, отдельный файл БД):**
  - вставка 100 / 1000 / 10 000 массивов: флаг успеха + время;
  - выгрузка и сортировка 100 случайных массивов — 3 прогона на БД с 100, 1000, 10 000 записями: флаг, общее время, среднее время на 1 массив;
  - очистка БД — 3 прогона для тех же размеров: флаг + время (при необходимости с повторным наполнением).
- **GUI:** с учётом эвристик UI/UX (статус, согласованность, минимализм, ошибки, справка и т.д.).
- **Функционал приложения:** регистрация, вход, ввод/генерация массива, сортировка алгоритмом из лаб. №2, сохранение, просмотр истории.

## 3. Используемые технологии (п. 2 и 4 методички)

### 3.1. SQLite (встраиваемая СУБД)

- **Суть:** одна библиотека, хранение в файле (в проекте: `sorting.db` в каталоге приложения).
- **Плюсы:** не нужен отдельный сервер, один файл, транзакции, SQL, кросс‑платформенность, простое развёртывание.
- **Минусы:** не классика для сильно многопользовательской нагрузки; сравнение с «большими» серверами — ограниченный набор сценариев администрирования.

### 3.2. Графика: C++/CLI, Windows Forms

- **Суть:** десктопное приложение, форма `MainForm`, управляющие элементы WinForms.
- **Плюсы:** быстрая визуальная разработка, встроенные контролы, стабильная интеграция с .NET.
- **Минусы:** привязка к Windows; кастомизация внешнего вида слабее, чем у WPF/веб-стеков.

## 4. Структура базы данных

### Таблица `users`

| Поле        | Тип     | Назначение        |
|------------|---------|-------------------|
| id         | INTEGER | PK, autoincrement |
| login      | TEXT    | уникальный логин  |
| password   | TEXT    | пароль (в учебной версии хранится как текст) |
| created_at | DATETIME| время создания    |

### Таблица `arrays`

| Поле        | Тип     | Назначение |
|------------|---------|------------|
| id         | INTEGER | PK         |
| user_id    | INTEGER | FK → `users.id` |
| original   | TEXT    | исходный массив (числа через пробел) |
| sorted     | TEXT    | отсортированный массив (через пробел) |
| created_at | DATETIME| момент сохранения |

Связь `ON DELETE CASCADE` — при удалении пользователя удаляются и его массивы.

## 6. Описание графического интерфейса, достоинства/недостатки, эвристики UI/UX (п. 4)

### 6.1. Достоинства (относительно WinForms-реализации `MainForm`)

1. **Логичная группировка:** три панели-карточки `groupAuth` (авторизация), `groupArray` (работа с массивом), `groupHistory` (история в БД) — зоны читаются сверху вниз, без «перемешанных» кнопок.  
2. **Простота и понятность:** поля с подписями «Логин:», «Пароль:», «Массив (через пробел):», кнопки с однозначным текстом.  
3. **Предзаполненное поле массива:** `txtArray` по умолчанию содержит тестовый набор `5 3 8 1 2 9 4 7 6` — удобно для быстрой проверки.  
4. **Понятные надписи кнопок:** «Войти», «Регистрация», «Сгенерировать», «Сортировать», «Сохранить в БД» и т.д.  
5. **Статусная строка `lblStatus`:** отображает состояние (подключение к SQLite, результат сортировки/сохранения, ошибки), цвет текста подчёркивает успех/ошибку.  
6. **Всплывающие сообщения `MessageBox`:** предупреждения при пустом массиве, отсутствии входа при сохранении, успех сохранения.  
7. **Естественный порядок:** сначала вход/регистрация, затем ввод/генерация и сортировка, затем при необходимости сохранение и просмотр истории.  
8. **Ограничение сохранения без авторизации:** если пользователь не вошёл (`currentUsername` пуст), **сохранение в БД не выполняется** — выводится предупреждение «Сначала войдите в систему!» (логическая блокировка; кнопка визуально доступна, но операция не проходит).  

### 6.2. Недостатки (что в учебной версии намеренно не реализовано или слабее, чем в «крупных» UI)

1. Нет **кастомных тем** оформления (как в Material/WPF) — стандартный стиль WinForms + небольшая кастомная отрисовка панелей.  
2. Нет **горячих клавиш** (F1 не привязан, Ctrl+S и т.п.).  
3. Нет **контекстного меню** в списке истории.  
4. Нет **экспорта** истории в файл.  
5. Нет **поиска/фильтра** по истории.  
6. **Пароли** в таблице `users` хранятся **в открытом виде** (для production нужен хэш + соль).  
7. Нет **блокировки учётной записи** после серии неверных паролей.  

### 6.3. Соответствие эвристикам Nilsen (кратко, п. 4 задания)

| Эвристика | Как реализовано в программе |
|-----------|----------------------------|
| Видимость статуса | `lblStatus` + цвет, сообщения о БД/сортировке/сохранении |
| Соответствие миру, управляемость, согласованность | Бытовая формулировка кнопок, сценарий «войти → обработать массив → сохранить» |
| Минимализм, гибкость (в меру) | Три сгруппированных блока, без лишних экранов |
| Помощь в диагностике ошибок | `MessageBox` с пояснениями, подсказки `ToolTip` на части кнопок |
| Справка | Кнопка **«Справка»** (`btnHelp`) → окно с инструкцией |

### 6.4. Структура главного окна (имена из `MainForm.h`)

**Главное окно** — `Form` с заголовком вроде «Сортировщик массивов с SQLite БД».

1. **Группа «Авторизация»** (панель `groupAuth`, заголовок `lblAuthTitle` «Авторизация»)  
   - `txtLogin` — логин; `txtPassword` — пароль; `chkShowPassword` — «Показать пароль».  
   - `btnLogin` — «Войти»; `btnRegister` — «Регистрация»; `btnLogout` — «Выйти»; `btnClearHistory` — «Очистить историю» (логика доступности зависит от входа).  

2. **Группа «Работа с массивом»** (`groupArray`, заголовок `lblArrayTitle` «Работа с массивом»)  
   - `txtArray` — `RichTextBox` с массивом (числа через пробел).  
   - `btnGenerate` — «Сгенерировать»; `btnSort` — «Сортировать»; `btnSave` — «Сохранить в БД».  

3. **Группа «История (SQLite)»** (`groupHistory`, заголовок `lblHistoryTitle` «История (SQLite БД)»)  
   - `listHistory` — список кратких записей о сохранениях; подписи `labelHistory` и т.д.  

4. **Низ формы**  
   - `lblStatus` — статусная строка.  
   - `btnHelp` — «Справка».

## 7. Выводы

- Реализовано **десктопное** приложение с **WinForms** и **SQLite** для хранения пар пользователей и пар «исходный/отсортированный» массив.
- Сортировка — **шейкерная** (класс `ShakerSort`, из лабораторной работы №2), вызывается из GUI и из тестов.
- Интеграционные сценарии вынесены в отдельный проект с **отдельной БД** `sorting_test.db`.
- Для **продакшена** важно: хэширование паролей, доработка UX (горячие клавиши, темы) — в учебной версии часть ограничений осознанно не реализовывалась.
